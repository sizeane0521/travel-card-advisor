## Context

目前 CalcPage 中，換算台幣金額（`≈ NT$X`）與一般說明文字使用相同字級與色彩，導致使用者在掃視時難以快速定位最重要的數字。信用卡卡片的回饋拆解字串（`NT$734 = 基本 NT$262 + 加碼 NT$315 + ...`）為一長串 inline text，在手機螢幕上容易斷行，影響可讀性。TripsPage 旅程卡片只顯示消費/回饋/筆數，幣別/匯率資訊無處呈現。

## Goals / Non-Goals

**Goals:**

- 建立統一的「金額數字 = 主要資訊」視覺規則：大字 + 高亮色
- 建立統一的「拆解說明 = 次要資訊」視覺規則：小字 + 低對比色
- 旅程卡片顯示幣別/匯率
- 修正 +刷卡 的 endDate 判斷 bug

**Non-Goals:**

- 不重新設計整體色彩系統或 design token
- 不改動回饋計算邏輯
- 不調整資料結構

## Decisions

### 換算台幣金額視覺強化

**決定**：`≈ NT$X` 改用 `text-2xl font-bold`，顏色改為 `#f2e8c9`（主文字白色），與輸入欄下方的說明文字拉開層級。

**理由**：使用者輸入日幣後，最想確認的就是台幣換算結果；目前 `text-sm + #c8a060` 視覺權重太低，容易被忽略。

**替代方案**：改用特殊底色 highlight — 否決，增加視覺噪音。

### 信用卡回饋資訊視覺層級

**決定**：將 `formatBreakdown` 輸出（長字串）拆成兩層：
1. **主要**：回饋總額 `NT$XXX` — `text-lg font-bold`，顏色 `#4ade80`
2. **次要**：各項拆解 — `text-xs`，顏色 `#9a7040`，逐項換行顯示

**理由**：使用者掃視卡片時，先確認「能賺多少」（總額），細項僅供驗證用。目前一長串混排，主次不清。

**替代方案**：折疊展開細項 — 否決，互動成本過高且細項有參考價值。

### 旅程卡片幣別/匯率

**決定**：若 `trip.exchangeRate` 存在，在卡片底部統計資訊行後增加一行小字：`{currency} · 匯率 {rate}`，顏色 `#9a7040`。

**理由**：給使用者快速確認旅程設定，不需要進入旅程才能看到。

### +刷卡 endDate 判斷修正

**決定**：`handleRecordWithCard` 中 line 185 的條件從 `activeTrip.endDate` 改為 `activeTrip.endDate && activeTrip.endDate <= todayStr()`，與渲染層的判斷邏輯一致。

**理由**：設了未來結束日期的旅程應仍可記帳；原條件只要 `endDate` 為 truthy 就阻擋，邏輯過於嚴格。

## Risks / Trade-offs

- [視覺調整] 放大換算金額後，若螢幕較小可能擠壓版面 → 現有 `max-w-lg` 容器已有足夠空間，風險低
- [拆解顯示] 由一行改多行後，卡片高度增加 → 可接受，資訊密度降低反而更好讀
